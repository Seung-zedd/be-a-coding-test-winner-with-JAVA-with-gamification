### 🧠 던전 통찰 (Dungeon Insights)

**1. Stock Prices: 인덱스 역추적과 후처리의 미학**

- **인덱스 저장:** 값 자체보다 '언제' 들어왔는지가 중요. `Stack<int[]>`에 `[index, price]`를 함께 담아, 나중에 `i - index`로 시간을 계산.
- **반응 조건 검증:** `while`문을 통해 현재 가격보다 비싼 이전 가격들을 모두 찾아내어 '생존 종료'를 선언함. 이는 `Daily Temperatures`와 같은 문제의 핵심 로직.
- **잔존 세력 소탕:** 루프가 끝난 후에도 스택에 남은 이들은 '끝까지 가격이 떨어지지 않은' 승리자들. 이들에게는 `(전체 길이 - 1) - index`라는 보상을 수여해야 함.

**2. Visit Length: 점(Node)에서 선(Edge)으로의 사고 전환**

- **양방향 경로 중복 제거:** (A→B)와 (B→A)를 동일시하기 위해 두 방향을 모두 `Set`에 넣고 2로 나누는 전술.

**3. 괄호 vs 주식 가격: if와 while의 결정적 차이**

- **단순 매칭 (Matching):** 괄호 문제는 현재와 스택 top의 1:1 관계만 확인하면 되므로 `if`문으로 충분.
- **단조 스택 (Monotonic):** 주식 가격이나 Daily Temperatures는 '현재'가 '과거의 여러 시점'에 영향을 줄 수 있으므로, 조건이 만족되는 한 계속해서 pop을 수행하는 `while`문의 폭풍이 필요함. 이것이 바로 $O(N^2)$을 $O(N)$으로 줄이는 마법.

**4. Next Greater Element: 단조 스택의 정수(Essence)**

- **인덱스 활용의 극대화:** 스택에 `int[] {index, value}`를 넣어도 좋지만, 사실 `index`만 넣어도 `numbers[index]`로 원본 값을 참조할 수 있어 메모리를 더 아낄 수 있음 (백만 개 데이터 처리 시 유리).
- **부등호의 미학:** 결국 주식 가격(단조 증가)과 뒷 큰수(단조 감소)는 `while`문의 부등호 방향만 다른 '쌍둥이' 문제. 본질적인 단조 스택의 매커니즘을 이해하면 부등호 하나로 수많은 유사 문제를 $O(N)$으로 썰어버릴 수 있음.

**5. 연산자 우선순위와 방어적 로직의 결합**

- **결합의 중요성:** `!st.isEmpty() && A || B`는 `( !st.isEmpty() && A ) || B`로 실행된다. 스택이 비어있는 상태에서 B가 실행되면 Exception이 터진다. 반드시 `!st.isEmpty() && (A || B)` 처럼 괄호로 안전하게 묶어야 한다.
- **문자열 회전의 정석:** `s.substring(x) + s.substring(0, x)`는 문자열의 $x$번째부터 끝까지를 앞으로 보내고, 처음부터 $x$전까지를 뒤로 붙여 완벽한 '왼쪽 회전'을 구현한다. 단일 문자(`charAt`)만 합치는 실수를 경계해야 한다.

**6. 모의고사: 추상화의 벽을 넘는 해시맵 인덱싱**

- **구조의 본질:** 해시맵의 `getValue()`로 반환된 배열은 결국 2차원 배열의 행(Row)과 같다. 이 배열에 `%` 연산자를 적용하는 순간, 복잡한 Map 구조는 친숙한 인덱스 기반 정답지로 변모한다.
- **시간 복잡도의 상수화:** $O(N \times M)$에서 $M$(수포자 수)이 상수일 때, 이는 알고리즘적으로 $O(N)$의 효율을 보장한다. 데이터의 규모가 커질수록 상수항을 제어하는 능력이 승패를 가른다.

**7. Matrix Multiplication: 3중 루프와 Dot Product의 필연성**

- **인덱스의 역할:** `i`는 `arr1`의 행을, `j`는 `arr2`의 열을 지정하며, `k`는 그 사이에서 실제 성분들을 맺어주는 가교 역할을 한다.
- **차원 일치의 법칙 ($c1 == r2$):** 행렬의 곱셈은 본질적으로 (행벡터) dot product (열벡터)이다. 내적이 성립하려면 두 벡터의 성분 개수가 일치해야 하므로, `arr1`의 열 개수와 `arr2`의 행 개수가 같아야만 3중 루프의 가장 안쪽인 `k`가 논리적으로 완결될 수 있다.
- **시간 복잡도의 직관:** 결과 행렬의 모든 칸($r1 \times c2$)을 채우기 위해 각각 $c1$(혹은 $r2$)번의 연산이 필요하므로 $O(N^3)$의 비용이 발생한다.

**8. Crane Claw Machine: 텍스트 분해와 시각화의 힘**

- **지문 해체:** 방대한 요구사항도 '조건'과 '제약'으로 나누어 주석으로 정리하면 더 이상 공포의 대상이 아니다.
- **인덱스 사상 (Mapping):** 크레인의 좌우 이동을 `moves[i] - 1` (Column)로, 인형을 찾는 과정을 `for(int i=0; i < r; i++)` (Row)로 시각화하여 '가장 위에 있는 인형'을 꺼내는 완벽한 동선을 설계.
- **연쇄 폭발 (LIFO):** `Stack.get(size-2)`와 `peek()`을 활용한 실시간 검증 로직으로, 인형이 쌓이는 즉시 폭발시키는 기민함을 보여줌.

**9. Table Edit: 우회와 영속성의 미학**

- **삭제가 아닌 우회:** `next[prev[k]] = next[k]` 연산은 데이터를 물리적으로 삭제하지 않는다. 단지 `prev[k]`번 노드가 `k`를 건너뛰고 `next[k]`를 바라보게 할 뿐이다. 이는 JPA의 `detach()`가 영속성 컨텍스트와의 연결만 끊는 것과 궤를 같이한다.
- **자생적 복구 정보:** `Z` 명령어가 $O(1)$인 이유는 삭제된 번호 `z` 자체가 자신의 `prev[z]`와 `next[z]` 정보를 여전히 '배열'이라는 불멸의 저장소에 간직하고 있기 때문이다. "내가 누구와 연결되었었는지"를 본체가 기억하고 있기에 가능한 기적이다.
- **k vs z의 주체성:** 모든 명령은 현재 위치 `k`를 기준으로 주변을 수정하지만, 오직 `Z`만은 복구되는 `z`를 주인공으로 하여 주변 이웃들을 다시 불러모은다.
- **StringBuilder의 생존성:** $1,000,000$개의 데이터를 `+` 연산자로 합치면 메모리 복사와 객체 생성의 늪($O(N^2)$)에 빠져 즉사한다. `StringBuilder`는 가변(Mutable) 버퍼를 사용하여 선형 시간($O(N)$)에 안전하게 결과를 도출한다.

**10. Josephus Problem: 큐 회전과 CQS의 조화**

- **회전의 미학:** `q.offer(q.poll())`은 원형 구조를 물리적인 이동 없이 논리적으로 완벽하게 구현한다. 인덱스 계산의 번거로움을 큐의 선입선출(FIFO) 특성으로 전이시킨 탁월한 전술.
- **CQS (Command-Query Separation):** `for-each`라는 조회 전용 정찰병을 보내면서 동시에 `poll`과 같은 명령(수정)을 내리면 안 된다는 원칙을 배웠다. `while` 루프 안에서 조회와 명령을 명확히 분리할 때 전장은 비로소 안정된다.
- **히든 비용의 발견:** 수학적으로 인덱스만 툭 바꾸는 것처럼 보여도, ADT의 구현체(ArrayList 등)에 따라 내부적으로 대규모 Shifting 작업($O(N)$)이 발생할 수 있음을 간파했다.

**11. Discount Promotion: TCP 슬라이딩 윈도우와 증분 업데이트의 미학**

- **캐시 기반 증분 업데이트:** 10일치 데이터를 매번 새로 세는 대신, '나가는 데이터(-)'와 '들어오는 데이터(+)'만 단 2번 연산하여 $O(N)$의 효율성을 달성함.
- **Map.equals()의 완결 조건:** 수량이 0이 된 품목을 `remove()` 함으로써, 맵의 '키 집합'을 정답지와 완벽히 일치시켜야 한다는 점이 핵심.
- **메모리 계층의 통찰:** 가운데 9개는 변하지 않는다는 점을 이용해 이전의 연산 결과를 재사용(Caching)하는 기법은 컴퓨터 네트워크의 Go-back-N 철학과 궤를 같이함.

**13. computeIfPresent: 조건부 소거의 미학**

- **null의 마법:** 람다식에서 `null`을 반환하면 키가 자동으로 삭제된다는 점을 이용해 `Map.equals()` 비교를 위해 필요한 '0인 키 제거'를 $O(1)$의 우아함으로 해결함.
- **안전한 수정:** 키가 존재할 때만 작동하므로 의도치 않은 데이터 삽입을 방지하는 방어적 코딩의 정수.

**14. Open Chat: 문자열 결합의 $O(N)$ vs $O(N^2)$ 진실**

- **누적의 함정:** `String A = ""`에 `A += B`를 루프 안에서 계속 반복하면, 객체가 점진적으로 거대해지며 1+2+3...+N의 복사 비용이 발생하여 **$O(N^2)$** 시간 초과 지옥에 빠진다.
- **독립 결합의 우아함:** 리스트에 넣을 때 `list.add(str_a + str_b)`처럼 임시 객체를 생성하는 것은, 매 루프마다 항상 아주 짧은 문자열 2개를 합치는 $O(1)$ 상수 비용만 든다. 이를 N번 반복해도 전체 비용은 **$O(N)$**으로 완벽하게 안전하다.
- **결론:** 거대한 하나의 도화지에 계속 붓칠을 해야 할 때만 `StringBuilder`를 꺼내고, 작게 잘라진 메시지 조각들을 배열/리스트에 차곡차곡 담을 때는 직관적인 `+` 연산자를 마음껏 믿고 휘둘러라.

**15. Open Chat 심화: 2-Pass 알고리즘과 CS 아키텍처의 결합**

- **RDT (TCP) vs 지연 평가 (Lazy Evaluation):** RDT(신뢰적 데이터 전송)의 FSM은 '상대방의 패킷을 확인'하는 즉각적인 **신뢰성(Reliability)**이 생명이므로 상태 변화 지연을 허용하지 않는 엄격한 1-Pass(실시간) 아키텍처다. 반면, 오픈 채팅방은 중간 과정이 어찌됐든 **최종 닉네임(Final Consistency)**만이 핵심이므로 출력을 뒤로 미루는 2-Pass 지연 평가가 최적의 무기가 된다!
- **JPA 쓰기 지연 (Transactional Write-Behind):** 엔티티 조작(`persist`, `update`)을 건건이 DB에 날리면(1-Pass) I/O 낭비가 심하다. 트랜잭션 종료 시점(commit)까지 해시맵 형태의 1차 캐시에 '최종 상태'만 모아둔 뒤, 마지막에 SQL을 한 번에 생성해내는(2-Pass) 원리와 완벽히 동일하다.
- **React Virtual DOM:** 상태 변경마다 실제 화면(DOM)을 리렌더링하는 것은 브라우저 입장에서 $O(N^2)$급 재난이다. 따라서, JS 객체인 가상 DOM에 최종 상태만 갱신(1-Pass)해 두고, 차이가 확정되면 한 번에 실제 DOM을 칠하는 지연 평가 로직이다.
- **결론:** '과거 소급'의 함정을 벗어나는 2-Pass 지연 평가는 단순한 코테 기술이 아니라, 현대 컴퓨터 공학에서 **가장 무거운 I/O 연산을 최적화하기 위해 고안된 위대한 철학**이다.
