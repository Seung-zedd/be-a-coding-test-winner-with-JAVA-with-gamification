# 🛠️ 논리 대장간 (LOGIC WORKSHOP)

이곳은 용사님의 사고의 균열을 발견하고, 더 날카로운 논리로 재련하는 연마실입니다. 모든 코딩 테스트 퀘스트의 '사고 진화 과정'을 기록합니다.

---

## 🛡️ [문제 27] 다단계 칫솔 판매 (Toothbrush Sales)

### 1단계: 초기 인식 (Intuition)

- "일단 판매원들의 총매출(`amount * 100`)을 `tallyMap`에 다 모으자."
- "그다음 트리 위로 10%씩 전파하며 깎아 나가면 2-Pass(집계 후 처리)로 깔끔하게 끝나겠지?"
- **의도:** 불필요한 트리 탐색 횟수를 줄여서 효율성을 높이려고 함.

### 2단계: 논리 조립 (Architecture)

- **Data Structure:** `parentMap` (추천인 관계), `moneyMap` (최종 수익 정산).
- **Process:**
  1. `tallyMap` 생성.
  2. 판매원 리스트를 돌며 부모에게 10% 상향 전파 (While loop).
  3. `moneyMap` 업데이트.

### 3단계: 실전 발화 (Verbatim Flow)

- "각 판매 데이터를 **'독립적으로 발생하는 이벤트'**로 규정합니다. 금융 도메인의 복잡한 절사 규칙에 매몰되지 않고, 각 이벤트가 발생할 때마다 트리를 거슬러 올라가며 10%의 수익을 상향 전파(Propagate)시키는 모델을 구축합니다."

### 4단계: 사고의 균열 & 교정 (Reflection)

- **🚨 균열 (The Crack):** **수학적 절사(Floor)의 함정.**
  - 9원짜리 10번 팔면, 매번 10%인 0.9원은 절사되어 추천인에게 가는 돈은 총 0원임.
  - 하지만 이걸 90원으로 묶어서 한 번에 전파하면, 10%인 9원이 추천인에게 가게 됨.
  - **결과:** 문제의 요구사항인 "판매 시마다 10%"와 어긋나며 테케 통과 실패.
- **🛠️ 교정 (The Fix):** '집계 후 전파'를 폐기하고, **'발생 즉시 전파'**로 회귀. 문자열 비교 시 `==` 대신 `.equals()`를 엄격히 사용하고, `if`문 분기 대신 **[통합 루프(Unified Loop)]**를 도입하여 `null` 안정성 확보.

### 5단계: 진화된 사고 (Evolution)

- **독립적 이벤트 전파 (Independent Event Propagation):**
  - 복잡한 도메인 지식에 매몰되는 대신, 문제를 **'독립적인 이벤트의 연속'**으로 치환하여 사고를 단순화하라. 각 이벤트가 발생할 때마다 상태를 전이시키는 모델이 가장 견고하다.
- **상태의 일관성:** 반복문이든 재귀든 '현재의 상태(Person, Money)'를 다음 단계로 온전히 전이시키는 것이 핵심이다.
- **Transactional vs Aggregated (결정적 통찰):**
  - 절사(Integer Division)가 존재하는 환경에서 '개별 정산 후 합산'과 '합산 후 정산'은 결과가 완전히 다르다. (예: 9원씩 10번 팔면 개별 정산 시 총 0원 전파).
  - **교훈:** 비즈니스 규칙이 '사건 발생 시'를 기준으로 되어 있다면, 함부로 Batch(일괄) 처리하여 데이터의 무결성을 훼손하지 마라.
- **철학:** "복합적인 예외 처리(`if`)는 로직의 일반성 부족을 의미한다. 루프 조건 자체에 예외를 흡수시켜 통합(Unify)하라."
