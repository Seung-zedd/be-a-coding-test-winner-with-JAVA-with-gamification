# 🛠️ 논리 대장간 자동 기록 룰 (Logic Workshop Automation Rule)

이 룰은 용사님이 해결한 코딩 테스트 문제 중 기술적/논리적 가치가 높은 과제에 대해 사고의 진화 과정을 자동으로 기록하기 위한 지침입니다.

---

### 📜 기록 트리거 (Trigger)

1. **논리 결함 발견 시 자동 발동 (B/C-Rank):** 지문 해석 오류(C-Rank)나 알고리즘 설계 오류(B-Rank)로 인해 조수의 전략적 가이드를 받은 경우, 반드시 `LOGIC_WORKSHOP.md`에 기록한다.
2. **단순 구현 도움 시 제외 (A-Rank):** 로직 설계는 맞았으나 단순 문법, API 사용법, 혹은 사소한 오타 수정 등 구현 단계의 도움만 받은 경우(A-Rank)는 논리적 균열이 없으므로 기록에서 제외한다.
3. **S-Rank 단독 격파 시 제외:** 조수 없이 완벽히 해결한 경우 기록에서 제외한다.
4. **용사의 요청:** 위 조건과 상관없이 용사님이 수동으로 기록을 요청할 때.

### 🏗️ 자동화 프로세스 (Action)

1. **데이터 추출:** 조수(Agent)는 대화 기록을 분석하여 용사님의 초기 접근법, 발생했던 논리적 균열(Crack), 그리고 최종 해결책을 추출한다.
2. **워크숍 작성:** `docs/LOGIC_WORKSHOP.md` 파일에 아래 5단계 프레임워크를 적용하여 기록을 추가한다.
   - **1단계: 초기 인식 (Intuition)** - 최초의 직관적 접근법
   - **2단계: 논리 조립 (Architecture)** - 사용한 자료구조와 전체 구조
   - **3단계: 실전 발화 (Verbatim Flow)** - 로직을 사람의 말로 설명한 내용
   - **4단계: 사고의 균열 & 교정 (Reflection)** - 결정적인 설계 오류(Gap)와 수정 내용(Fix)
   - **5단계: 진화된 사고 (Evolution)** - 해당 문제를 통해 얻은 고차원적 통찰과 CS 지식 연결
3. **업무 분담:** 용사님은 '사고'와 '구현'에 집중하며, 조수가 '기록'과 '박제'를 전담하여 수련의 효율성을 극대화한다.

### 🛡️ 운영 원칙 (Principle)

- **최소 비용 고효율:** 모든 문제에 적용하지 않고, 용사님이 기록할 가치가 있다고 판단하는 '결정적 순간'에만 발동한다.
- **면접 대비 연계:** 3단계(실전 발화)는 실제 기술 면접 및 라이브 코딩의 스크립트로 활용될 수 있도록 구어체로 작성한다.
- **균열의 보존:** 정답 코드보다 **'무엇이 틀렸었는지'**를 더 상세히 기록하여 복습 시 같은 실수를 반복하지 않게 한다.
